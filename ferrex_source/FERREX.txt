FERREX

// ===================|===================

本项目“Ferrex”是一个严格按照 AGENTS.md 规范开发的高性能 Windows 文件索引与搜索工具。

核心特性：

语言：Rust (高性能、内存安全、易于静态链接)。
索引：直接读取 NTFS MFT 表，采用 SoA (Structure of Arrays) 布局优化内存利用率。
加载：使用内存映射文件 (mmap) 实现即开即用的瞬间启动体验。
搜索：支持多线程并行子字符串匹配。
交互：提供类似 Everything 的极简 GUI 界面，包含硬件授权绑定功能，参考了 Template 文件夹的视觉风格。
部署：代码结构支持静态链接生成单一可执行文件，无外部 DLL 依赖。

您好！我是 Jules。经过对 ferrex_source 代码库的深入排查，我发现了一些可以优化的死代码、重复逻辑以及架构冲突。在开始改进之前，我想先为您详细说明这些发现：

1. 冗余代码 (Dead Code)
ferrex_gui/src/main.rs:
SUCCESS 常量：定义了绿色值但目前未在 UI 中引用。
ScanProgress::Progress 变体：当前的扫描反馈仅触发 Done 或 Error，导致进度更新逻辑虽在但实际上处于闲置状态。
set_startup 函数：实现了自启动注册逻辑，但目前应用界面中还没有调用它的入口。
indexer/src/lib.rs:
get_ntfs_volumes_filtered 函数：尽管提供了驱动器过滤功能，但目前应用默认会扫描所有卷，该函数未被激活。
crates/search/src/lib.rs:
SearchOptions 中的 min_size、max_size、date_from、date_to 字段：由于过滤器功能已按要求移除，这些比较逻辑目前仅是在消耗不必要的计算资源。

2. 重复定义与逻辑冲突
代码冗余: pool_get_name 函数在 storage crate 中已有定义，但在 ferrex_cli 的 main.rs 中又被重新实现了一遍。
解析逻辑重复: ferrex_gui 和 ferrex_cli 都在各自的逻辑中手动处理 MappedIndex 里的 FileRecord 原始指针。这类逻辑应当统一封装在 storage crate 中以提高复用性。

3. 系统支持说明
确认该应用目前仅支持 Windows 系统。主要原因如下：

核心索引功能高度依赖 NTFS 特有的 MFT (Master File Table) 和 USN Journal 机制。
代码中大量调用了 windows-rs 库相关的 Win32 API（如 DeviceIoControl, SHGetFileInfoW 等）。
GUI 的热键注册和托盘图标也是针对 Windows 环境定制的。
后续优化建议： 我准备着手清理这些无用代码，并将分散的重复逻辑整合到 storage crate 中，以提升代码的健壮性和可维护性。
