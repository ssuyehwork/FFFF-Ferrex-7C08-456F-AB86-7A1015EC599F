数据库逻辑强制要求 file_id_128 字段永不为空。对于尚未计算哈希的文件，必须物理生成格式为 'FRN:[Volume]:[FRN]' 的 Fallback ID，以防止 JOIN 操作产生笛卡尔积或 category_items 主键冲突。

系统内置分类的数据加载通过 ItemRepo::getPathsBySystemType(type) 统一获取，不应与通用搜索逻辑 searchByKeyword 耦合。

在 Qt 6 环境下，QTreeView 子类（如 DropTreeView）的命中测试（indexAt）必须通过 viewport()->mapFrom(this, pos) 进行坐标转换，以确保拖拽命中位置的准确性。

当文件从 Fallback ID 更新为真实 128-bit 哈希时，必须同步级联更新 category_items 表中的关联记录，确保分类关系在哈希更新后不丢失。

项目核心底层逻辑以基于 WinAPI 提取的 128-bit File ID (FRN) 作为唯一锚点，用于在文件移动或重命名时追踪元数据，严禁回归到单纯的路径绑定。

数据库已启用 WAL (Write-Ahead Logging) 模式，以优化高频并发写入性能。

数据库性能加固：category_items 表必须在 category_id 字段上建立索引，items 表必须在 file_id_128, rating, color, pinned 字段建立索引，以支撑 2.4M 级大数据量下的实时查询。

当前项目架构采用仓储模式（Repository Pattern），通过 CategoryRepo、ItemRepo、FolderRepo 等类封装数据持久化逻辑，实现业务逻辑与底层存储的解耦，显著优于旧版本的单体上帝类（God Class）模式。

数据库管理类（Database）提供 getThreadDatabase() 接口，遵循 Qt 数据库跨线程“单连接单线程”的最佳实践，确保在多线程文件系统追踪及同步时的线程安全。

侧边栏顶级架构：物理重构为“系统部分”与“自定义部分”两大顶级根节点。“系统部分”包含8个功能项（全部、未分类、未标签、今日、昨日、最近、标签、回收站）；“自定义部分”包含“快速访问”与“我的分类”两个嵌套子模块。

视觉与交互规范：所有自定义分类、物理文件夹及归类菜单项统一强制使用 folder_filled 图标；模块标题（如“系统部分”、“自定义部分”）锁定为白色（#FFFFFF）；新建分类（手动或拖拽）的默认颜色固定为深灰色（#555555）。系统内置项维持原有的语义化彩色图标。

## 界面架构规范标准 (基于 FramelessDialog)

### 1. 核心架构理念
*   **物理级无边框方案**：摒弃系统原生边框，采用 `Qt::FramelessWindowHint` 结合 `Qt::WA_TranslucentBackground` 实现。
*   **0 延时瞬时显隐**：对于 ToolTip 等辅助窗口，应改用 `Qt::Window` 结合 `Qt::WindowDoesNotAcceptFocus` 方案，以规避操作系统强制附加的淡入淡出动画，确保交互的瞬时反馈。
*   **扁平化布局架构**：ArcMeta 采用极致扁平化设计，**全面废除外发光阴影效果**。分层布局应以简化为原则，确保圆角处视觉纯净，不产生锯齿。

### 2. 关键视觉参数
*   **边距标准**：**全面移除 20px 阴影边距**，窗口布局应紧贴物理边缘 or 仅保留必要的 UI 呼吸间距。
*   **窗口圆角 (Window Radius)**：对话框主体容器 (`DialogContainer`) 统一采用 **6px** 或 **12px**。
*   **交互颜色**：
    *   **统一悬停色**：采用 `#3e3e42`，适用于按钮、菜单项等所有可点击组件。
    *   **激活状态色**：主色调使用 `#007ACC` 或 `#4A90E2`。
    *   **安全/成功状态**：采用 `#00A650`。
    *   **警告/紧急状态**：采用 `#E74C3C`。

### 3. 按钮风格规范 (Button Style)
*   **标准功能按钮**（标题栏内置）：
    *   **标准尺寸**：统一锁定为 **24x24** px。
    *   **布局边距**：`margin` 设为 **2px**。
    *   **圆角设置**：统一采用 **4px** 圆角。
    *   **图标规范**：仅使用 **SVG** 格式，建议尺寸 **14x14** px。
*   **对话框特定规范 (FramelessDialog)**：
    *   **按钮去留**：除主窗口外，弹出式对话框（如输入框、确认框）**仅保留“关闭按钮”**。
    *   **关闭按钮视觉**：高亮背景必须**持续锁定为红色** (`#E81123`)，图标使用 white 以增强对比。
    *   **交互颜色**（通用）：悬停态为 `#3e3e42`，按下态为 `rgba(255, 255, 255, 0.15)`。

### 4. 自定义标题栏 / 工具栏规范 (TitleBar & ToolBar)
*   **核心理念：极致精简**。杜绝臃肿，追求极致的扁平化与紧凑感。
*   **尺寸极限限制**：
    *   **主工具栏高度**：**绝对不可超过 36px**。
    *   **对话框高度规范**：文本输入类对话框 (`FramelessInputDialog`) 统一采用扁平化高度，标准设定为 **210px**。
    *   **垂直布局 (Vertical)**：宽度 **绝对不可超过 36px**。
*   **按钮尺寸**：内部功能按钮统一锁定为 **24x24 px**，以确保在 36px 的容器内留有呼吸感且不显臃肿。
*   **布局细节**：采用紧凑的 `QHBoxLayout` 或 `QVBoxLayout`，间距及 `padding` 应压缩至最小值，优先保证有效内容的展示。

### 5. 交互行为规范
*   **边缘缩放感应区 (Resize Zone)**：
    *   **感应宽度**：严格限制在 **12px** 范围内。
    *   **隔离原则**：感应区严禁侵入内容容器内部，以防用户点击边缘按钮时被识别为缩放操作而导致点击失效。
*   **两段式关闭逻辑**：对于包含输入框的对话框（如 `FramelessInputDialog`），`Esc` 键的首选功能是“清空内容”，仅当内容已空时再次按下才执行“关闭窗口”。
*   **焦点锁定**：弹出式确认框（如 `FramelessMessageBox`）在显示时必须强制将焦点锁定在“确定”按钮上，确保回车操作的连贯性。

## 傻逼逻辑架构审计记录 (核心技术硬伤清单)

### 1. 同步引擎 (SyncEngine) 逻辑断层
*   **现状**：`SyncEngine::runFullScan` 与 `runIncrementalSync` 内部逻辑已被清空/废弃。
*   **死穴**：由于彻底移除 `.am_meta.json`，系统目前完全丧失了 **离线变更自愈能力**。若在程序关闭期间发生物理文件位移或重命名，下次启动时没有任何机制能主动追平这些变更。这是实现 2.4M 海量数据稳健性的最大绊脚石。

### 2. UI 线程同步刷新悖论
*   **现状**：`CategoryModel::refresh()` 在 UI 主线程同步执行。
*   **死穴**：刷新逻辑中调用的 `CategoryRepo::getSystemCounts()` 包含 8 个复杂的 `COUNT(DISTINCT)` 数据库查询。在 2.4M 级大数据量下，这种同步设计会导致 UI 线程被物理锁定数百毫秒，是产生界面卡顿感、破坏“秒开”体验的根源。
*   **改进方向**：必须将计数逻辑彻底异步化，通过局部刷新 (CountRole) 补全数据。

### 3. “掩耳盗铃”式搜索去重
*   **现状**：在 `ItemRepo` 搜索中大量使用 `GROUP BY file_id_128` 配合 `LIMIT 1000`。
*   **死穴**：对数百万行数据进行全文搜索后再做聚合运算非常昂贵。目前的 `LIMIT` 只是掩盖了查询低效的本质，并没有从索引或架构层面解决去重带来的算力透支。

### 4. 路径处理与检索模式的“双标”矛盾
*   **现状**：`MetadataManager` 保护路径大小写敏感，但 `ItemRepo` 使用 `LIKE %keyword%` 进行暴力模糊匹配。
*   **死穴**：这种匹配模式无法触发 SQLite 索引，在大数据量下会导致 Full Table Scan（全表扫描）。内存缓存的优势在最关键的检索环节完全没有体现，导致性能上限被底层暴力检索锁死。

## 核心架构红线与开发偏好 (禁止脑补准则)

### 1. 架构基石：128-bit File ID 中心化绑定
*   **唯一识别码**：整个数据库的逻辑架构必须围绕物理文件的 **128-bit File ID** 展开。1 个数据库 ID 必须严格对应一个唯一的物理 File ID。
*   **多对多解耦绑定**：所有用户操作产生的元数据（包括但不限于：打分、标签、颜色、锁定/置顶状态、分类关联）**必须对该物理 File ID 进行多对多绑定**。
*   **架构目的**：确保文件在物理层面移动或重命名后，元数据绑定关系依然稳固且能通过 ID 自动追平。

### 2. 检索铁律：全量索引驱动
*   **工业级性能要求**：在大数据量（2.4M 级）环境下，禁止使用任何形式的暴力全表扫描。
*   **检索逻辑**：对打分、标签、颜色等属性的搜索，必须且只能按照数据库物理索引进行高效查询。

### 3. 开发流程红线：严禁脑补与擅自修改
*   **禁止盲猜**：Jules 严禁对用户意图进行任何形式的“脑补”或“猜测”。任何逻辑模糊点必须以提问形式向用户确认，并提供可选方案。
*   **绝对授权流程**：必须严格执行 **“探讨意图 -> 达成深度共识 -> 明确获取【授权修改】字样指令”** 的三段式状态机。在未获得明确授权前，严禁触碰任何一行逻辑代码。
*   **行为定性**：脑补带来的代码蔓延与破坏等同于对工程环境的“投毒”，必须坚决杜绝。
