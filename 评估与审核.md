


这份代码实现了一个类似 Everything 的高性能本地文件搜索工具（基于 `egui` 和 NTFS MFT 扫描），整体架构和 UI 设计很不错。但在**核心业务逻辑、性能细节和跨平台/环境兼容性**上存在几个较为严重的逻辑错误。

以下是代码中存在的逻辑错误分析，按严重程度排序：

### 1. 严重：列表渲染的“假滚动”与数据截断问题
**位置：** `draw_results_list` 函数中
```rust
let results_to_show  = self.results.len().min(200);

ScrollArea::vertical()
    .auto_shrink([false, false])
    .show(ui, |ui| {
        for idx in 0..results_to_show { ... }
    });
```
**问题：** 
代码虽然查出了最多 5000 条结果，但 UI 渲染循环被**强行截断在 200 条**。这导致即使用户往下滚动，也永远只能看到前 200 个文件，剩下的 4800 个文件就像消失了一样。
**修复建议：** 
不应该截断数组，而应使用 `egui` 提供的虚拟化滚动 API `ScrollArea::show_rows`，这样即使展示 10 万条数据也不卡顿：
```rust
ScrollArea::vertical().show_rows(ui, 30.0, self.results.len(), |ui, row_range| {
    for idx in row_range {
        let result = &self.results[idx];
        // 渲染逻辑...
    }
});
```

### 2. 严重：WMI 硬件校验因编码问题 100% 失败
**位置：** `verify_hardware_wmi` 函数中
```rust
if let Ok(output) = cmd.output() {
    let text = String::from_utf8_lossy(&output.stdout);
    // ... eq_ignore_ascii_case 判断
}
```
**问题：** 
在 Windows 中，当通过管道或 `cmd.output()` 捕获 `wmic` 的输出时，其默认编码是含有 BOM 的 **UTF-16LE (UCS-2)**。`String::from_utf8_lossy` 强行解析会导致文本中夹杂大量 `\0`（空字符）并将 BOM 变成乱码 ``（例如：`P\0r\0o\0c\0...`）。后续的 `trim()` 无法去除中间的 `\0`，这会导致 `whitelist` 的比对永远失败，**所有设备都会被拦截在“未授权的硬件设备”界面**。
**修复建议：** 
需要对 Windows 下的 UTF-16LE 进行处理，或过滤掉 Null 字节：
```rust
// 简单修复：将宽字符的空字节去除，并替换乱码
let raw_text = String::from_utf8_lossy(&output.stdout);
let text = raw_text.replace('\0', "").replace('\u{FFFD}', ""); 
```

### 3. 严重：USN 日志监听导致的数据假同步（Desync）
**位置：** `process_usn_events` 函数中
```rust
UsnEvent::Renamed { old_frn, .. } | UsnEvent::Modified { frn: old_frn } => {
    store.path_cache.pop(&old_frn);
}
```
**问题：** 
当文件发生重命名或移动时，代码仅仅从 `path_cache`（全路径缓存）中移除了它。但是！计算新路径所依赖的 `Arc<LoadedIndex>`（包含 `string_pool`, `parent_frns`, `name_offsets` 等）是**只读且未被更新的**。
这会导致：
1. `resolve_path` 重新计算路径时，仍旧读取到内存中旧的名字和旧的父节点，算出来的路径和改名之前一模一样。
2. 搜索新文件名搜不到，搜索旧文件名依然会出现。
**修复建议：** 
仅凭目前的静态 `IndexStore` 设计无法做到真正的 USN 同步更新。若要实现，必须将 `IndexStore` 的核心数组设计为支持内部可变性（如 `RwLock<Vec<u32>>` 等），并在 USN 回调中真正修改字符串池和父节点引用。

### 4. 逻辑瑕疵：修改时间的时区显示错误
**位置：** `format_timestamp` 函数中
```rust
use chrono::{TimeZone, Utc};
if let Some(dt) = Utc.timestamp_opt(secs, 0).single() {
    return dt.format("%Y-%m-%d %H:%M").to_string();
}
```
**问题：** 
文件系统的原始时间通常是 UTC 时间戳。代码中使用了 `Utc` 进行格式化，这意味着中国用户（+8区）看到的文件修改时间会比实际时间**早 8 个小时**。
**修复建议：** 
改用本地时区 `Local`：
```rust
use chrono::{TimeZone, Local};
if let Some(dt) = Local.timestamp_opt(secs, 0).single() { ... }
```

### 5. 性能隐患：排序导致 UI 线程严重阻塞
**位置：** `apply_sort` 函数中
```rust
SortColumn::Name => self.results.sort_by(|a, b| { 
    let c = a.name.to_lowercase().cmp(&b.name.to_lowercase()); 
    // ...
}),
```
**问题：** 
Rust 的 `sort_by` 复杂度是 $O(N \log N)$。如果匹配了 5000 条数据，在闭包内部调用 `to_lowercase()` 会导致约 **50000 ~ 60000 次堆内存分配（String）**。用户点击“名称”表头排序时，UI 会发生明显的卡顿。
**修复建议：** 
不要在排序闭包中分配字符串。可以使用针对 ASCII 大小写的忽略对比（如果仅需粗略英文忽略），或者使用底层提供的 API 避免拷贝：
```rust
// 无堆分配的大小写忽略排序方案 (支持 ASCII)
let c = a.name.cmp_ignore_ascii_case(&b.name);
```
*(注意：若追求完善的多语言不敏感排序，应在缓存层缓存小写特征，而非在 sort 循环内实时计算。)*

### 6. 数据导出：CSV 导出导致格式错乱
**位置：** `export_selected_csv` 函数中
```rust
content.push_str(&format!("{},{},{},{},{}\n", r.name, r.full_path, r.size, ...));
```
**问题：** 
如果用户的文件名或路径中包含逗号（比如 `Report, 2026.pdf`），将直接破坏 CSV 的列对齐。
**修复建议：** 
应该为字符串字段加上双引号进行转义：
```rust
content.push_str(&format!("\"{}\",\"{}\",{},{},{}\n", ...));
```

### 7. UI 交互：空文件大小显示歧义
**位置：** `format_size` 函数中
```rust
if bytes == 0 { return "—".to_string(); } 
```
**问题：** 
文件大小若恰好为 0 字节，它会返回 `"—"`。而在 UI 渲染逻辑中，目录（文件夹）的大小也是显示为 `"—"`。这让用户在视觉上无法区分 0 字节的空文件和文件夹。
**修复建议：** 
0 字节就是 0 字节，取消特判：
```rust
if bytes == 0 { return "0 B".to_string(); }
```

在上一轮针对明显严重缺陷的审核基础上，如果继续深入挖掘每个功能的底层逻辑、生命周期管理以及 UX 设计，代码中还存在以下**中高危级别的逻辑漏洞和不合理的交互设计**：

### 1. 致命缺陷：后台全局热键 (Alt+Space) 在隐藏时彻底失效
**位置：** `handle_hotkey` 函数与 `update` 生命周期
```rust
fn update(&mut self, ctx: &egui::Context, _frame: &mut eframe::Frame) {
    // ...
    #[cfg(windows)]
    self.handle_hotkey(ctx); // <-- 在这里轮询热键消息
}
```
**问题：** 
`egui` (底层基于 `winit`) 采用的是按需渲染（Reactive Rendering）机制。**当窗口被最小化或隐藏到托盘时，渲染循环会挂起，`update` 函数将不再被周期性调用**。
这就意味着，当用户隐藏窗口后，`handle_hotkey` 将停止轮询 `PeekMessageW`，你的全局热键 `Alt+Space` 会彻底瘫痪，再也无法唤醒程序。
**修复建议：** 
不要在 `update` 中处理全局热键。既然你已经在 `setup_tray` 中单独开辟了一个后台线程来维持托盘事件，应该**把全局热键的注册和消息循环 (`GetMessageW`) 转移到那个专门的后台脱管线程中**。

### 2. 逻辑断链：新硬盘扫描完毕后，数据成了“幽灵”
**位置：** `spawn_scan` 与 `handle_scan_progress`
```rust
ScanProgress::Done => { 
    self.is_scanning = false; 
    drop_rx = true;
    self.status_text = "扫描完成".to_string(); 
}
```
**问题：** 
当遇到未建立索引的新驱动器时，程序调用 `spawn_scan` 扫描 MFT，并在完毕后写入了 `{drive}_drive.idx` 文件，然后发送 `Done` 消息。
然而，`handle_scan_progress` 收到 `Done` 之后，**仅仅是修改了 UI 状态，却没有将刚生成的 `.idx` 文件加载进内存 (`self.stores`)**。这就导致用户满怀期待地等完了扫描，却发现根本搜不到该盘的文件，必须手动重启软件才能生效。
**修复建议：** 
在处理 `Done` 事件时，应该重新触发加载逻辑：
```rust
ScanProgress::Done => { 
    self.is_scanning = false; 
    drop_rx = true;
    self.status_text = "扫描完成".to_string();
    self.load_volumes(); // 重新加载新生成的索引
    self.run_search(ctx); // 刷新搜索结果
}
```

### 3. 并发漏洞：多驱动器扫描状态互相踩踏
**位置：** `load_volumes` 函数
```rust
let (tx, rx) = std::sync::mpsc::channel();
self.scan_rx = Some(rx); // <--- 单一 Option 存储
self.is_scanning = true;
spawn_scan(vol, tx);
```
**问题：** 
如果用户插上了一个包含两个新分区的移动硬盘（或者初次启动时 C/D/E 盘全都没有索引），`load_volumes` 的循环会连续调用三次 `spawn_scan`。
但是，由于 `self.scan_rx` 只是一个普通的 `Option`，**后面的 channel 接收器会直接覆盖前面的**。这会导致只有最后启动的那个扫描线程的进度能回传到 UI，前几个盘的扫描彻底变成“失控的僵尸线程”，就算扫完了 UI 也不予理会。
**修复建议：** 
若要支持多盘并发扫描，应将 `scan_rx` 改为 `Option<Receiver<ScanProgress>>` 配合共享的 `mpsc::channel`，多个 `tx` clone 给各个扫描线程；或者将 `is_scanning` 和队列系统化，改成串行扫描。

### 4. 性能阻滞：在搜索主循环中同步读取系统图标
**位置：** `run_search` 函数
```rust
for rec_idx in matches.into_iter().take(5000) {
    let name = pool_get_name(...);
    // 致命点：UI 线程里同步去调 Windows API 解析图标
    self.icons.get_for_path(ctx, &name, store.index.flags[rec_idx] & 0x10 != 0);
    // ...
}
```
**问题：** 
虽然 `get_for_path` 内部有按照扩展名进行缓存，但是在初次运行或命中冷门后缀时，它会触发 `load_system_icon` 调用重量级的 `SHGetFileInfoW` 和 GDI API。
如果你搜到了大量的不同后缀文件，或者恰巧有几十个 `.exe`（即使只靠扩展名查），主线程会被这段代码死死卡住。**搜索结果应该在 10 毫秒内出来，但图标加载可能会花掉数百毫秒甚至数秒，导致搜索时出现“卡死”感。**
**修复建议：** 
不要在 `run_search`（获取数据层）中加载图标。**图标获取必须是惰性（Lazy）的**。应该在 `draw_results_list`（UI 渲染层）中，**仅针对当前滚动条视野内**可见的十几条结果调用 `icons.get_for_path`。

### 5. 核心体验缺失：缺失“双击打开文件”操作
**位置：** `draw_results_list`
**问题：** 
作为一款文件搜索工具，用户习惯是“双击列表项直接打开文件”。但是当前代码中，对于 `response` 的交互判定只有：
- `response.clicked()` -> 选中该行
- `response.secondary_clicked()` -> 弹出右键菜单
代码中根本没有绑定双击事件，用户想打开文件只能被迫使用反人类的 `右键 -> 打开文件`。
**修复建议：** 
加入双击判定：
```rust
if response.double_clicked() {
    open_file(&result.full_path);
} else if response.clicked() { 
    new_selected = Some(idx); 
}
```

### 6. 交互逻辑错误：Shift 键连选功能残废
**位置：** `draw_results_list` 行选择逻辑中
```rust
} else if modifiers.shift {
    self.selected_rows.insert(idx);
}
```
**问题：** 
在 Windows/Mac 的标准交互逻辑中，`Ctrl + Click` 才是单点追加选择（你代码里的 `.command` 表现正确），而 `Shift + Click` 应当是**区间选择**（即选中上次点击的项目与这次点击的项目之间的所有行）。
当前代码将 `Shift` 的行为实现成了和 `Ctrl` 摸鱼一样的单点追加，违背了用户的肌肉记忆。
**修复建议：** 
在 `FerrexApp` 的状态中增加一个 `last_clicked_row: Option<usize>`。在检测到 `modifiers.shift` 时，将 `last_clicked_row` 到当前 `idx` 范围内的所有索引遍历插入到 `self.selected_rows` 中。

---

**总结：**
整体框架非常清晰，MFT 扫描加 `egui` 是很好的技术选型。但在落地细节上，**“后台托盘/热键休眠问题”** 和 **“新盘扫描并发与反馈覆盖”** 是必须优先解决的 Bug；而 **“懒加载渲染区图标”** 和 **“加入双击打开文件”** 则是让这个产品能真正用起来的核心要点。

这份代码相比上一版有了极大的提升！你成功解决了长列表渲染、多盘并发扫描、USN日志覆盖（Overlay）以及WMI硬件码解析的痛点，整体架构已经非常接近一个成熟的商业级本地搜索工具。

但在进一步深挖生命周期交互和文件系统逻辑后，仍然存在 **4个关键的逻辑缺陷**，它们会直接影响核心功能的使用（如文件打不开、图标显示错乱、崩溃闪退）：

### 1. 严重漏洞：路径拼接错误导致“文件永远打不开”
**位置：** `VolumeStore::resolve_live_path` 函数中
```rust
let path = format!("{}:\\{}", self.drive, parts.join("\\"));
```
**问题分析：**
Windows 系统下，获取驱动器的名字通常是 `"C:"` 或 `"C:\ "`。如果你直接使用 `"{}:\\{}"`，当 `self.drive` 是 `"C:"` 时，拼接出来的结果会变成 **`"C::\xxx\yyy"`**。如果包含斜杠则是 `"C:\:\xxx"`。
这会导致在列表中显示非法路径，双击或右键“打开文件”时，操作系统完全无法识别，毫无反应。
**修复建议：**
安全地去除驱动器末尾可能存在的反斜杠，并直接拼接：
```rust
let clean_drive = self.drive.trim_end_matches('\\');
// 如果 clean_drive 只是 "C" 补上冒号，如果是 "C:" 则直接用
let drive_prefix = if clean_drive.ends_with(':') { 
    clean_drive.to_string() 
} else { 
    format!("{}:", clean_drive) 
};
let path = format!("{}\\{}", drive_prefix, parts.join("\\"));
```

### 2. 逻辑脏数据：多次搜索导致“幽灵选择”与越界崩溃
**位置：** `run_search` 函数末尾
```rust
self.results = all_results;
self.apply_sort();
// 遗漏了清理旧状态！
```
**问题分析：**
如果在一次搜索中，你按 `Shift` 选中了第 50 行到第 100 行。接着你在搜索框里输入了新关键词，匹配结果只剩下 5 条。
此时 `self.results` 的长度变成了 5，但 `self.selected_rows` 里**依然保存着 50~100 的索引**！
1. 界面上会出现选中状态的错乱。
2. 如果此时你点击“导出CSV”，代码尝试访问 `self.results.get(50)` 尚能通过 `if let` 避免崩溃。
3. 如果执行 `Shift` 连选，可能会导致跨域的灾难性插入。
**修复建议：**
每次执行新的查询并更新结果后，必须重置所有的选择上下文：
```rust
self.results = all_results;
self.apply_sort();
self.selected_rows.clear();         // 新增：清除旧的选择记录
self.last_clicked_row = None;       // 新增：清除连选锚点
self.context_menu_row = None;       // 新增：收起可能遗留的右键菜单
```

### 3. UI 视觉 Bug：所有的 EXE 共享同一个图标
**位置：** `IconCache::get_for_path` 函数中
```rust
let ext = if is_dir { "folder".to_string() } else {
    path.rsplit('.').next().unwrap_or("").to_lowercase()
};
if let Some(texture) = self.textures.get(&ext) {
    return texture.into();
}
```
**问题分析：**
你采用了按“扩展名（ext）”来做图标缓存，这对于 `.txt`、`.mp4` 这种系统通用图标非常聪明，极大提高了性能。
**但是，`.exe` 和 `.ico` 是例外！** 每一个可执行文件（Chrome、QQ、微信）都有自己独立的图标。按当前逻辑，由于它们扩展名都是 `"exe"`，列表里所有的程序**都会显示你渲染的第一个 EXE 的图标**（比如全部变成 Chrome 的图标）。
**修复建议：**
对这两种特殊扩展名，将缓存的 Key 从“扩展名”改为“完整路径”：
```rust
let ext = if is_dir { 
    "folder".to_string() 
} else {
    path.rsplit('.').next().unwrap_or("").to_lowercase()
};

// 如果是 exe 或 ico，必须使用完整路径作为缓存 Key
let cache_key = match ext.as_str() {
    "exe" | "ico" => path.to_lowercase(),
    _ => ext.clone(),
};

if let Some(texture) = self.textures.get(&cache_key) {
    return texture.into();
}

if let Some(texture) = self.load_system_icon(ctx, path, is_dir) {
    self.textures.insert(cache_key.clone(), texture.clone());
    return (&self.textures[&cache_key]).into();
}
```

### 4. 隐蔽的资源泄漏：后台全局热键注销失败
**位置：** `Drop for FerrexApp`
```rust
impl Drop for FerrexApp {
    fn drop(&mut self) {
        #[cfg(windows)]
        unsafe { let _ = UnregisterHotKey(HWND(0), 1001); } // 此处运行在 UI 主线程
    }
}
```
**问题分析：**
Windows 系统底层 API 规定：`RegisterHotKey` 和 `UnregisterHotKey` 具有**线程亲和性（Thread Affinity）**。
你在上一步修改中，非常正确地把 `RegisterHotKey` 放在了负责托盘的独立后台线程里。但是，当程序退出走 `Drop` 时，它是在 `egui` 的主线程执行注销的！这就导致注销指令100%失败，热键被孤立（虽然进程结束后操作系统会兜底回收，但属于不严谨的代码行为）。
**修复建议：**
不需要在主程序的 `Drop` 里注销。由于你开启了托盘守护线程，直接让托盘线程在捕获到退出信号时自己注销即可：
```rust
// 修改 setup_tray 托盘线程中的 quit_id 分支：
} else if event.id == quit_id {
    should_exit.store(true, Ordering::SeqCst);
    ctx.send_viewport_cmd(ViewportCommand::Close);
    ctx.request_repaint();
    
    // 自主清理线程创建的资源
    #[cfg(windows)]
    unsafe { let _ = UnregisterHotKey(HWND(0), 1001); }
    
    std::thread::spawn(|| {
        std::thread::sleep(std::time::Duration::from_secs(1));
        std::process::exit(0);
    });
}
```
并将 `impl Drop for FerrexApp` 整个删除即可。

### 为什么这样改就能解决？
1. `.with_menu_on_left_click(false)` 告诉操作系统，托盘图标左键点击时**不要**托管给系统原生的菜单渲染机制。

3. 你的后台线程捕捉到这个 `Left` 事件后，执行 `ViewportCommand::Visible(true)` 和 `Focus`，主界面就会如期完美弹出了！

是的，**右键单击列表确实会弹出菜单**，因为代码中捕获了右键点击事件 (`response.secondary_clicked()`) 并触发了弹出逻辑。

**但是！在目前的实现中存在一个非常滑稽的视觉 Bug（“菜单瞬移”）：**

仔细看你手动实现右键菜单定位的这一段代码：
```rust
let menu_pos = ui.input(|i| i.pointer.interact_pos().unwrap_or_default());
Area::new(Id::new("ctx_menu")).fixed_pos(menu_pos) ...
```
在 `egui` 这样的即时渲染（Immediate Mode）UI 框架中，代码是**每帧执行**的。
1. 当你按住右键的瞬间，`interact_pos()` 返回了真实的鼠标坐标，菜单在你右击的地方弹出。
2. 当你**松开右键**（或者只要没发生点击动作），`interact_pos()` 会返回 `None`。
3. 此时 `.unwrap_or_default()` 会返回 `Pos2::new(0.0, 0.0)`。
4. **结果就是：菜单会在你松开鼠标的瞬间，瞬间飞（瞬移）到窗口的左上角 (0,0) 处！**

### 1. 致命缺陷（静默崩溃）：USN 通道堆积导致 OOM 与 UI 假死
**位置：** `process_usn_events` 与 `update` 生命周期
**问题分析：**
`egui` 默认运行在**按需渲染 (Reactive) 模式**。如果用户把程序挂在后台，或者放在一边不碰鼠标键盘，`update` 函数是**完全不会执行**的！
此时，后台的 NTFS USN 监听线程会疯狂地将磁盘变化（可能每秒几百条）塞入 `mpsc::channel`。
- 如果闲置 5 个小时，通道里会积压几千万个 `UsnEvent`，狂吃几个 GB 的内存导致 OOM 崩溃。
- 如果此时用户碰了一下鼠标唤醒了窗口，UI 线程的 `while let Ok(event) = rx.try_recv()` 会被迫一口气处理几千万条日志，导致**界面瞬间卡死数秒甚至几十秒**。
**修复建议：**
必须强迫 `egui` 定期苏醒，即使没有鼠标操作也要抽干（Drain）日志通道：
```rust
// 在 eframe::App 的 update 函数最顶部加上这句：
fn update(&mut self, ctx: &egui::Context, _frame: &mut eframe::Frame) {
    // 强制每 200ms 至少唤醒一次抽干消息队列，防止内存爆炸
    ctx.request_repaint_after(Duration::from_millis(200)); 
    
    // ... 原来的代码
}
```

### 2. 严重逻辑 Bug：生成的路径多出一个 `.\`
**位置：** `VolumeStore::resolve_live_path`
```rust
let (name, parent) = ...
parts.push(name);
if current_frn == 5 || parent == 0 || current_frn == parent { break; }
```
**问题分析：**
在 NTFS 文件系统中，根目录（如 `C:\`）的 MFT FRN 固定为 `5`，且它的系统内部名字是 `.`（一个点）。
按照你当前的拼接逻辑，溯源到 FRN = 5 时，会把 `.` push 进去。
这会导致所有文件的路径拼接出来都变成：**`C:\.\Windows\System32\cmd.exe`**。这个多余的 `.\` 极其难看且不专业。
**修复建议：**
在 push 之前，忽略根目录的内部名字：
```rust
let (name, parent) = ...;

// 新增过滤：只有当不是根目录(5)时，才把名字加入路径层级
if current_frn != 5 {
    parts.push(name);
}

if current_frn == 5 || parent == 0 || current_frn == parent { break; }
```

这三个问题非常典型，尤其后两个涉及到了 **Windows 底层 API 传参规则**和 **Rust 内存生命周期（野指针）**的经典坑。

以下是详细的原因分析和修复方案：

### 1. 为什么托盘菜单背景色始终是白色？
这是因为 `tray_icon` 这个库使用的是 **Windows 原生系统菜单 (Win32 API: `CreatePopupMenu`)**。
* `egui` 负责绘制的主界面是利用图形库（如 OpenGL/DirectX/Vulkan）画出来的，你可以随意控制它的颜色。
* 但是，托盘图标右键弹出的菜单，是由 **Windows 操作系统全权接管并渲染的**。它不属于 `egui`，因此你的深色主题对它无效。它**严格跟随你的 Windows 系统的全局应用颜色设置**（设置 -> 个性化 -> 颜色 -> 选择默认应用模式）。
* **结论**：除非你的 Windows 系统全局处于“深色模式”，否则它永远是白色的。要想强制给原生托盘菜单改色，必须使用 C++ 级别的“自绘菜单 (Owner-draw Menu)” Hook，目前 Rust 的 `tray_icon` 并不支持。建议直接接受这个原生设定（就像微信、QQ 的托盘菜单一样）。

---

### 2. 修复“在资源管理器中定位”失败
**原因分析：**
Rust 的 `Command` 为了安全，会自动给带有空格的参数加上双引号。
代码 `cmd.args(["/select,", path])` 最终在底层执行时，会被 Rust 拼接成了：
`explorer.exe "/select," "H:\AHK 可用\..."`
但是！Windows 的 `explorer.exe` 解析器非常古板，它不认识被拆开的引号，导致它去寻找一个错误的文件名，从而弹窗报错。

**修复方案：**
必须使用 Windows 特有的 `.raw_arg()`，绕过 Rust 的自动加引号机制，把参数完整地传给 explorer。

将 `reveal_in_explorer` 函数修改为：
```rust
fn reveal_in_explorer(path: &str) {
    let mut cmd = std::process::Command::new("explorer");
    #[cfg(windows)]
    {
        use std::os::windows::process::CommandExt;
        cmd.creation_flags(0x08000000); // CREATE_NO_WINDOW
        // 使用 raw_arg 原样传递参数，explorer 必须要求 "/select," 后面紧跟被双引号包裹的路径
        cmd.raw_arg(format!("/select,\"{}\"", path));
    }
    #[cfg(not(windows))]
    {
        cmd.arg("/select,");
        cmd.arg(path);
    }
    let _ = cmd.spawn();
}
```

### 3. 修复“属性”选项失败（经典野指针 Bug）
**原因分析：**
在原代码的 `open_properties` 中：
`lpFile: windows::core::PCWSTR(HSTRING::from(path).as_ptr())`
这是 Rust FFI (C语言互操作) 中最致命的陷阱：`HSTRING::from(path)` 产生了一个**临时变量**，它在 `.as_ptr()` 提取出指针后，立马就被释放销毁了！
导致传给 Windows 系统的指针变成了一个**野指针（指向已释放的内存）**。系统顺着指针读出来的是一堆乱码（如 `H:\AHK@定位行`），自然会报错“找不到文件”。

**修复方案：**
必须先将 `HSTRING` 绑定到独立的局部变量上，保证它们在 `ShellExecuteExW` 执行完之前一直存活。

将 `open_properties` 函数修改为：
```rust
#[allow(unused_variables)]
fn open_properties(path: &str) {
    #[cfg(windows)]
    unsafe {
        use windows::core::HSTRING;
        // 1. 将字符串绑定到变量，维持生命周期
        let verb = HSTRING::from("properties");
        let file = HSTRING::from(path);
        
        let mut info = SHELLEXECUTEINFOW {
            cbSize: std::mem::size_of::<SHELLEXECUTEINFOW>() as u32,
            // 2. 传递安全存活的指针
            lpVerb: windows::core::PCWSTR(verb.as_ptr()),
            lpFile: windows::core::PCWSTR(file.as_ptr()),
            nShow: 1, // SW_SHOWNORMAL
            fMask: SEE_MASK_INVOKEIDLIST,
            ..Default::default()
        };
        let _ = ShellExecuteExW(&mut info);
    }
}
```
